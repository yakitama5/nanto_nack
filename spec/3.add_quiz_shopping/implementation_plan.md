# 🚀 NantoNack クイズ機能 実装計画 (Quiz Feature Implementation Plan)

本計画は、基盤パッケージ（`system`, `app_main`）が構築済みであることを前提とし、`quiz_core` の拡張および個別のクイズカテゴリ（例: `quizzes/shopping`）を追加実装するための詳細なステップを定義します。

## Phase 1: クイズ共通基盤の拡張 (`packages/quiz_core`)

全クイズで使い回す「NantoNackならではのゲームシステム」を共通化します。

1. **サウンド＆ハプティクス制御の実装:**
   * BGMは実装せず、操作のフィードバックに特化したクラスを作成。
   * `playSuccessFeedback()`: 正解ルートの操作時（コトッという軽い振動＋小気味よい音）。
   * `playErrorFeedback()`: ミスタップ時（ブブッという重い振動＋鈍いエラー音）。
2. **解読不能文字（文字化け）システムの構築:**
   * `slang` で定義した日本語テキストに対し、特定のカスタムフォント（記号化フォント）を適用する `UnreadableText` ウィジェットを作成。
   * **チュートリアル演出:** 起動時は標準フォント、数秒後に `TweenAnimationBuilder` などを用いて解読不能フォントへスゥーッと変化するアニメーション機能を内包させる。
3. **ゲーム内共通UIコンポーネントの作成:**
   * **カットイン演出:** クイズ開始直後に全画面を暗転させ、「MISSION」と「制限時間」を数秒間デカデカと表示して消えるオーバーレイWidget。
   * **フローティングバー:** 画面上部に浮遊するピル型のバー。通常時は「⏳ 00:45 | 💡 ヒント | 🛒 お題」をコンパクトに表示し、タップで詳細が展開するWidget。
   * **リザルト画面:** 評価（S, A, B, C）、獲得スコア、および「UI/UXの種明かし解説」を表示する共通レイアウト。
4. **共通ドメインロジック (Score & Evaluation):**
   * **スコア計算式:** `(基本pt) ＋ (残り時間×ボーナスpt) － (ミスタップ回数×減点pt) － (ヒント使用ペナルティpt)` のロジック。
   * スコアに基づくランク付け（S〜C）の算出ロジック。

## Phase 2: カテゴリパッケージの構築 (`packages/quizzes/shopping`)

特定のアプリ（架空のECサイトなど）のガワと、そのステージ群を定義します。

1. **ドメイン層と静的データの定義:**
   * `ShoppingCategory` 固有のエンティティ（`Cart`, `Product` 等）を作成。
   * 各ステージ（Stage 0〜N）のマスターデータ（お題テキスト、制限時間、基本スコア、正解のプロダクトID、種明かしの解説文）を定義。
2. **ステージ選択画面の実装:**
   * 縦スクロールまたはグリッドでステージ一覧を表示。
   * `system`（Drift）から取得したクリア状況（`isCleared`）を参照し、未クリアの次のステージ以外はロック（🔒）表示にする順次解放UIの実装。
3. **架空のECアプリUI (背面レイヤー) の構築:**
   * フルスクリーンで表示するベースUI（ボトムナビゲーション、検索バー、商品グリッドなど）を作成。
   * テキストには `quiz_core` の `UnreadableText` を適用。

## Phase 3: クイズロジックと画面の統合 (各ステージの実装)

「ミッションをどうクリアするか・失敗するか」を個別ステージに組み込みます。

1. **状態管理 (Riverpod Notifier):**
   * `QuizState`: 残り時間（Timer）、ミスタップ回数、ヒント使用フラグ、現在のカート状態などを管理。
2. **ヒント機能の実装:**
   * フローティングバーの「💡ヒント」タップ時、スコア減点のペナルティを加算。
   * 状態を更新し、正解となるWidget（例：水の商品カードやカートアイコン）をハイライト表示させる、または対象の `UnreadableText` を標準フォント（読める日本語）に戻す処理を実装。
3. **正誤判定ロジックの組み込み:**
   * **クリア判定 (即時判定):** カートに条件のアイテムが入った状態で、「購入ボタン」をタップした瞬間にタイマーを停止し、クリア（リザルト画面へ遷移）。
   * **ミスタップ/失敗判定:** 指定と違う商品を購入確定してしまった場合はエラーフィードバックを鳴らし、ミスタップ回数を加算（または即座にゲームオーバー）。
   * 時間切れ（タイムアップ）処理。
4. **画面のスタッキング (UI統合):**
   * `Stack` ウィジェットを使用し、最背面に「架空のECアプリ」、その手前に「フローティングバー」、最前面に「カットイン演出（開始時のみ）」を配置して画面を完成させる。

## Phase 4: データ保存と連携

クイズの結果をシステムに反映します。

1. **リザルトから基盤への保存:**
   * リザルト画面で「次へ」または「ホームに戻る」を押した際、`system` パッケージのDrift Repositoryを呼び出し、スコアとクリア状況（`isCleared: true`）をローカルDBに保存。

## Phase 5: Testing（品質保証と自動テスト）

クイズ特有のロジック、UIレイアウト、およびユーザーの一連のプレイフローに対するテスト設計です。

1. **Unit Test (`test`):**
   * スコア計算ロジック（残り時間、ミスタップ、ヒント使用のペナルティ等）が正しく計算されるか。
   * ランク判定（S〜C）の境界値テスト。
2. **Widget Test (`flutter_test`):**
   * ボタンタップで正しくクリア判定・ミスタップ判定のイベントが発火するか。
3. **Golden Test (`alchemist`):**
   * 架空のECアプリ画面全体、および解読不能フォント（文字化け）が意図した通りに適用されているかの視覚的テスト。
   * `Stack` を用いたフローティングバーと背面レイヤーの重なり具合（各デバイス幅でのレイアウト崩れ防止）。
4. **E2E Test (`patrol`):**
   * ステージ選択 → カットイン演出待機 → アイテムをカートに入れる → ヒントを使用する → 購入アクション → リザルト画面でのスコア確認 → Driftへのデータ保存までの一連の通しテスト。

------

## 🛠 実装の具体的なステップ

1. **[Core/Testing]** `quiz_core` パッケージにスコア計算やランク付けのドメインロジックを作成し、**境界値のUnit Testを記述・通過させる**。
2. **[Core]** サウンド・ハプティクス制御と、文字化けアニメーション（`UnreadableText`）を実装。
3. **[Category]** `quizzes/shopping` パッケージを作成し、ステージ情報（マスターデータ）を定義。
4. **[Category/Testing]** ECアプリのベースUIとフローティングバーを構築し、**`alchemist` でデバイスサイズごとのGolden Testを実装する**。
5. **[Category/Logic]** Riverpodを用いてタイマー、ヒント状態、カート状態を管理するNotifierを作成し、状態変化のロジックをテスト。
6. **[Integration]** 画面のスタッキング（ECアプリ + フローティングバー + カットイン）を行い、リザルト画面でのDrift保存処理を結合。
7. **[Testing]** `patrol` を用いて、プレイヤーの操作（タップ・待機）を模倣した一連のE2E Testを記述し、クイズフロー全体が破綻していないかを確認。
